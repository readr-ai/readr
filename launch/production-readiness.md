# Readr — Production-Readiness Audit (Product Hunt launch)

Date: 2026-07-08 · Auditor: automated code audit · Basis: repo at `5ee8393` ("Product Hunt launch prep")

Every finding below cites the code that was actually read. Grading:
🔴 = launch blocker (breaks in a real user's hands, or a materially false public claim) ·
🟡 = ship with caveat (disclose, disable, or accept) · 🟢 = verified solid.

## 🔴 Launch blockers

### 1. Provider selection is never persisted — and never set when a key is saved. "Ask the book" appears broken on first run and after every relaunch.

**Failure scenario (first-day story):** User pastes an Anthropic API key in Settings → Save. Key lands in the Keychain, card shows "Connected". They select text → Ask → **"Connect an AI provider in settings to ask questions."** Same message greets every user who *did* get it working, on the next app launch.

**Why:**
- The active provider lives only in memory: `ProviderManager._selection` (`Sources/ReadrKit/LLM/ProviderManager.swift:40`), initialized to `nil` (`AppModel.swift:41-44` constructs `ProviderManager(store:factory:)` with no selection). `ProviderSelection` is `Codable` "persisted (by the UI, elsewhere)" (`ProviderManager.swift:3-4`) — but a repo-wide grep for `ProviderSelection`/`setActive`/`makeActive` shows **no code ever persists it** (no `UserDefaults`/`AppStorage` key exists for it; the only `@AppStorage` keys are `readingTheme` and `lastHighlightColor`).
- Saving an API key does not activate the provider: `SettingsModel.saveAPIKey` only writes the Keychain (`App/Settings/SettingsModel.swift:35-44`). The only path to a selection is the model `Picker`'s `set` closure (`App/Settings/ProviderSettingsView.swift:216-218`), which fires **only on change** — and the picker already *displays* the default model (`ProviderSettingsView.swift:210-213`), so a user who saves a key and closes Settings never triggers it.
- `AppModel.makeAskService()` returns `nil` when `activeProvider()` is `nil` (`App/AppModel.swift:596-599` → `ProviderManager.activeProvider()` line 93 `guard let selection ... return nil`), producing the "Connect an AI provider" error (`App/Ask/AskViewModel.swift:35-37`). Same for Article Studio (`App/Article/ArticleViewModel.swift:45-47`).

**Smallest viable fix:** (a) in `SettingsModel.saveAPIKey` / `signIn`, call `manager.setActive(kind: kind)` after a successful save; (b) persist the selection — `ProviderSelection` is Codable and non-secret, so a `UserDefaults` JSON blob read in `AppModel.init` and written in `setActive` is enough (~15 lines, unit-testable in `ProviderManagerTests`).

### 2. "Ask the book" with an OpenAI key silently omits the book — the model answers from the selection alone and will hallucinate.

**Failure scenario:** User connects OpenAI (`gpt-4.1`), opens a normal-sized book (< ~76k estimated tokens = 0.6 × 128k budget), asks "Why did the captain leave?". The router picks Tier 1 (whole book)… but the request that reaches OpenAI contains only the system prompt, the ~480-char selection anchor, and the question. The model confidently invents an answer. No error, no signal — the flagship "grounded in the whole book" promise (README.md:14-16, 24-26) is silently false on this path.

**Why:** `AdaptiveContextStrategy` gates the book text on caching support: `cacheableSystemPrefix: provider.supportsPromptCaching ? book.fullText : nil` (`Sources/ReadrKit/Context/ContextStrategy.swift:82`), while the Tier-1 routing check only tests `!provider.isLocal && estimatedTokenCount <= budget` (line 73). `OpenAIProvider` hard-codes `supportsPromptCaching: false` (`Sources/ReadrKit/LLM/Providers/OpenAIProvider.swift:27`) — so OpenAI + small book = whole-book tier with **no book**. The provider would handle it fine: `OpenAIProvider.encodeBody` maps `cacheableSystemPrefix` to a plain leading system message (`OpenAIProvider.swift:84-86`).

**Test gap that hid it:** `ContextStrategyTests` couples caching to locality — `supportsPromptCaching: !isLocal` (`Tests/ReadrKitTests/ContextStrategyTests.swift:19`) — so the (remote, non-caching) combination is never exercised.

**Smallest viable fix:** one line — always set `cacheableSystemPrefix: book.fullText` in the Tier-1 branch (each provider already decides how to encode it; Anthropic adds `cache_control`, OpenAI/local send it as a system message). Add one test with `isLocal: false, supportsPromptCaching: false` asserting the request contains the book text.

### 3. "Sign in with ChatGPT subscription" is unverified end-to-end and structurally unlikely to work — and even if it did, the token dies with no refresh path.

**Failure scenario:** First-run user clicks "Sign in with subscription" on the ChatGPT card (`App/Settings/ProviderSettingsView.swift:110-126`). The browser OAuth against `auth.openai.com` with the public Codex client id may complete (`Sources/ReadrKit/Auth/OAuthClient.swift:26-32`), but the resulting access token is then sent as `Bearer` to **`https://api.openai.com/v1/chat/completions`** (`Sources/ReadrKit/LLM/Providers/OpenAIProvider.swift:11` + `:70-75`). Codex-subscription OAuth tokens are not platform API keys (the Codex flow exchanges them or talks to the ChatGPT backend), so the first ask is expected to 401 — surfaced only as a generic "The operation couldn't be completed" (see 🟡-1). The user's takeaway on launch day: "the AI doesn't work."

**Second, independent failure:** token refresh is implemented and tested (`OAuthClient.refresh`, `Sources/ReadrKit/Auth/OAuthClient.swift:129-139`; `Tests/ReadrKitTests/OAuthClientTests.swift:106-142`) but **never called from the app** — a repo grep shows zero call sites for `refresh(` outside `OAuthClient`/tests, and `Credentials.isExpired` (`Sources/ReadrKit/LLM/Auth.swift:8-13`) is never read anywhere. When the access token expires (hours), every request 401s until the user manually Disconnects and re-signs-in. `docs/ROADMAP.md:36-37` confirms both: "Verify … endpoints (placeholder today)" mindset carries here — "Manual J5 walk on a Mac with a real provider; token-refresh-on-expiry wiring" is unchecked.

**Smallest viable mitigation for today:** hide the "Sign in with subscription" button (return `nil` for `.openAI` in `SettingsModel.oauthConfig(for:)`, `App/Settings/SettingsModel.swift:76-85` — one line, the UI already keys off it via `supportsOAuth`) or label it "beta / may not work with your plan". BYO API key — the documented default path (`docs/AUTH.md:49-53`) — is solid and stays. Proper fix post-launch: verify the token audience against the real endpoint and wire `isExpired`→`refresh` into `DefaultProviderFactory`/`ProviderManager.activeProvider()`.

### 4. README architecture claims are false: Readium and SQLite/sqlite-vec are not in the product.

**Failure scenario:** Launch-day HN/PH readers open the repo, read "**Readium Swift toolkit** for EPUB/PDF rendering & annotations" and "**SQLite** (`sqlite-vec` + FTS5) for the on-device RAG index" (README.md:45-46), then find `Package.swift` has **zero dependencies** (`Package.swift:13-17` — comment says "added as the corresponding milestones land"), `project.yml` has Readium **commented out** as a "later milestone" (`project.yml:22-25`; only ZIPFoundation is real, `project.yml:18-19`), and the RAG index is explicitly in-memory ("SQLite/sqlite-vec persistence is a later milestone", `Sources/ReadrKit/RAG/HybridRAGIndex.swift:5-7`; unchecked in `docs/ROADMAP.md:46,64`). On a launch built on "privacy-first, open source, look at the code," being caught overstating the stack is the top credibility risk.

**Smallest viable fix:** reword the two README bullets to what actually ships — "custom EPUB/PDF parsing (ZIPFoundation + PDFKit/native SwiftUI rendering)" and "in-memory hybrid BM25+vector RAG index (SQLite/sqlite-vec persistence on the roadmap)". *Note: README.md is being edited concurrently by other agents — flagging, not touching.*

## 🟡 Ship-with-caveat

1. **Provider/network errors are user-hostile.** No type in the codebase conforms to `LocalizedError` (repo grep: zero matches), and stream failures throw `HTTPError.status(code, body: "")` with the body deliberately empty (`Sources/ReadrKit/Networking/HTTPClient.swift:70-72`). The UI dutifully shows `error.localizedDescription` (`App/Ask/AskViewModel.swift:63`, `App/Article/ArticleViewModel.swift:78`) — which for these enums renders as *"The operation couldn't be completed. (ReadrKit.HTTPError error 0.)"* for a 401 (bad key), 429 (rate limit), or 529. Nothing hangs or crashes — the guard at `HTTPClient.swift:70` correctly aborts before streaming, and mid-stream drops finish the stream with the error, leaving the partial answer plus the message — but the message is not actionable. *Smallest fix:* one `LocalizedError` extension on `HTTPError`/`AuthError`/`ProviderManager.ProviderError` mapping 401→"check your API key", 429→"rate limited — try again shortly", etc.

2. **Abandoned OAuth sign-in wedges the Settings button.** `OAuthCoordinator.signIn` awaits a continuation resumed only by the loopback callback (`App/Auth/OAuthCoordinator.swift:30-40`). If the user closes the browser tab without completing (no redirect ever arrives), the continuation never resumes: `isSigningIn` stays `true` and the sign-in button shows a spinner, disabled, until app restart (`SettingsModel.swift:48-49`, `ProviderSettingsView.swift:114-125`). Also, the loopback listener is created with `NWListener(using: .tcp, on: nwPort)` with no interface restriction (`App/Auth/LoopbackHTTPServer.swift:31`) — it listens on all interfaces, contradicting `docs/AUTH.md:60` ("binds 127.0.0.1"). Low practical risk (`state` is validated, single-use), moot if 🔴-3's button is hidden.

3. **`library.json` contains the full text of every book and is rewritten whole on every mutation.** `FileLibraryStore.State.books` stores complete `Book` values including all chapter text (`Sources/ReadrKit/Library/FileLibraryStore.swift:8-17`; `Book.chapters[].text`, `Sources/ReadrKit/Models/Book.swift:76-83`), and `persist()` re-encodes the entire state on the caller's thread — the MainActor for every UI path (`FileLibraryStore.swift:43-46`, `AppModel` is `@MainActor`). The team knows: covers were moved out of the JSON precisely because "the store rewrites its whole JSON on every position save" (`App/AppModel.swift:287-290`), and page-turn saves are debounced 1 s (`App/Views/ReaderView.swift:188-197`). But every highlight/bookmark/state write is immediate — with a shelf of a few large EPUBs (tens of MB of text), each highlight means a multi-MB JSON encode + write on the main thread. Fine for a day-one library of 1–5 books; will jank for power users. Roadmap already tracks the SwiftData/GRDB replacement (`docs/ROADMAP.md:27`). *Disclose as known limitation; consider moving `persist()` off-main as a cheap stopgap.*

4. **Persistence failures are silently swallowed.** Disk-full or write errors vanish: `try? store.savePosition` (`App/AppModel.swift:443`), `try? store.saveBookState` (`:461`), highlight-add failure returns `nil` with a comment "nothing to add" (`:573-576`), and every bookmark/PDF-highlight mutation is `try?` (`:505-536`). No data corruption (writes are atomic, see 🟢-1) — but a user on a full disk loses annotations with zero feedback. *Acceptable for launch; add an error toast later.*

5. **Citations only appear for large books / local models.** README promises answers "with full book context and **source citations**" (README.md:24-26), but citations are populated only on the retrieval tier — the whole-book tier explicitly returns none (`Sources/ReadrKit/Context/ContextStrategy.swift:85` + `:28-29` "Empty for the whole-book tier"; test `testWholeBookTierHasNoCitations`, `ContextStrategyTests.swift:63-73`). Most hosted-model users with normal-sized books will never see a citation pill. *Disclose or soften the claim.*

6. **Token estimate is chars/4 — under-counts dense/CJK text.** `estimateTokens` (`Sources/ReadrKit/Reader/BookParser.swift:26-28`) can under-estimate a real 250k-token book enough to route it Tier-1 into a 200k window → provider 400 ("prompt too long") → generic error (🟡-1). The 0.6 budget fraction (`ContextStrategy.swift:60-62`) absorbs most of this; genuine risk only for non-Latin-script books near the boundary.

7. **Local (Ollama) model list is hard-coded and never validated.** Catalog offers exactly `llama3` and `qwen2.5` (`Sources/ReadrKit/LLM/ProviderCatalog.swift:56-71`). A user running Ollama with only e.g. `mistral` pulled gets a 404 from `/api/chat` → generic error; Ollama not installed → connection refused → generic error. The local card shows "Connected" regardless (`ProviderManager.isConfigured` returns `true` for `.local` unconditionally, `ProviderManager.swift:72-77`). *Disclose "requires Ollama with llama3 or qwen2.5 pulled".*

8. **No real-provider smoke test has ever run.** `docs/ROADMAP.md:37,47,54,65` — every "manual walk on a Mac with a real provider" item is unchecked; README.md:8-9 discloses this honestly ("real-provider smoke tests" remain). Positives verified against current docs: the Anthropic model IDs `claude-opus-4-8`, `claude-sonnet-4-6`, `claude-haiku-4-5` (`ProviderCatalog.swift:14-34`) are **valid, active IDs**, `anthropic-version: 2023-06-01` (`AnthropicProvider.swift:14`) is the current header, the Messages-API body shape with `system` blocks + `cache_control: ephemeral` (`AnthropicProvider.swift:87-128`) matches the documented API, and the 200k `contextBudget` is conservative (actual window 1M — errs safe). `gpt-4.1` is a real OpenAI model; its body uses deprecated-but-accepted `max_tokens` (`OpenAIProvider.swift:94`). One caching nit: Opus 4.8's minimum cacheable prefix is 4096 tokens, so very small books pay full price silently — cost, not correctness.

## 🟢 Verified solid

1. **Reading-position/annotation persistence is crash-safe.** Writes are atomic (`data.write(to:options:.atomic)`, `FileLibraryStore.swift:45`) — a crash or power loss mid-write leaves the previous file intact. A corrupt/truncated `library.json` on launch is **quarantined to `library.json.corrupt`, not overwritten** (`FileLibraryStore.swift:30-40`), so user data stays recoverable. All access is `NSLock`-guarded (`:20,49…`); multiple macOS reader windows share one store instance via one `AppModel`. Pre-v2 files decode via optional v2 fields (`:13-17`), covered by `V2BackwardCompatibilityTests` / `FileLibraryStoreV2Tests`. Position saves are debounced but **flushed on chapter change and onDisappear** ("closing the reader must never lose the last position", `ReaderView.swift:172-197`); worst-case loss is one sub-1-second scroll delta on force-quit.

2. **No crash-on-user-action force-unwraps.** Repo-wide grep for `try!`/`fatalError`/`as!` across `Sources/` and `App/`: the only hits are compile-time-constant `URL(string:)!` for hardcoded endpoints and `ProviderCatalog.first!` on non-empty static arrays (`ProviderCatalog.swift:90-92`), plus a `fatalError` in `PKCE.sha256` that is unreachable on Apple platforms (behind `#if canImport(CryptoKit)`, `Sources/ReadrKit/Auth/PKCE.swift:59-68`).

3. **Import edge cases are handled with actionable UI.** Encrypted/locked PDFs → `.drmProtected` (`App/Parsers/PDFKitBookParser.swift:18-20`); scanned PDFs with no text → "no extractable text — this may be a scanned PDF" (`:29-31`); unopenable EPUB zip → `.corrupted` (`App/Parsers/EPUBFileParser.swift:38-42`); DRM `encryption.xml` rejected (ROADMAP M1, `EPUBBookParserTests`). All parser errors map to friendly copy (`AppModel.swift:635-644`) shown in an alert (`LibraryShellView.swift:63-69`). Import copies/renders off the main actor (`AppModel.swift:274-282`). Huge files: text extraction is memory-bound (whole book in RAM) but linear — no unbounded recursion or quadratic paths found in `Chunker`/`XHTMLTextExtractor`.

4. **"Sign in with Claude" correctly does not exist.** `SettingsModel.oauthConfig(for: .anthropic)` returns `nil` with a ToS rationale (`SettingsModel.swift:76-85`); the Anthropic card offers API key only (badge "API key", `ProviderSettingsView.swift:146-153`). The unused `anthropicUnsupported` config is a deliberately inert marker with a screaming client id (`OAuthClient.swift:34-45`). Matches `docs/AUTH.md:41-46`. No first-run user can click a dead Claude OAuth button.

5. **Secrets hygiene holds.** Credentials live only in `KeychainCredentialStore` (`kSecClassGenericPassword`, `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` — device-only, excluded from backups and iCloud Keychain sync, `Sources/ReadrKit/Auth/KeychainCredentialStore.swift`); grep confirms `UserDefaults` is used only for `readingTheme` and `lastHighlightColor`. PKCE is real S256 with `SecRandomCopyBytes` (`PKCE.swift:24-27,71-78`); OAuth `state` is checked and consent-denial surfaced (`OAuthClient.swift:91-109`). `PrivacyAuditTests` cover the local path: the on-device retrieval pipeline takes no `HTTPClient`, and the local provider is asserted to contact loopback only (never an external API).

6. **Performance routing does what's claimed.** A 300k-token book against a 200k-budget model routes to retrieval — threshold is `0.6 × contextBudget` = 120k tokens (`ContextStrategy.swift:60-73`), tested for both directions (`ContextStrategyTests.swift:24-44`) and local-always-retrieval (`:92-101`). The in-memory RAG rebuild on first open per launch is cheap by construction: hash-trick embeddings (FNV-1a into 256-dim, no ML model, `LocalEmbeddingProvider.swift:23-45`) over ~1.2k-char chunks — a full-length novel is a few thousand chunks ≈ a few MB of vectors, built in a background task (`ReaderView.swift:199-201`, `AppModel.ensureIndexed`), reused across questions within a run (`HybridRAGIndex.isBuilt`, `AppModel.swift:613-616`). Retrieval quality is BM25-dominated (hash embeddings are quasi-lexical) — adequate, but not "semantic search" if anyone asks.

7. **Streaming plumbing is disciplined.** Every provider propagates cancellation (`continuation.onTermination → task.cancel()`, `AnthropicProvider.swift:57`, `OpenAIProvider.swift:55`, `LocalLLMProvider.swift:55`, `AskService.swift:65`); Ask guards re-entrant submits (`AskViewModel.swift:33-34`); Article Studio cancels on dismiss and handles empty-output and no-highlights cases without an LLM call (`ArticleViewModel.swift:35-79`; `ArticleComposerTests`). Non-2xx aborts before any tokens render (`HTTPClient.swift:70-72`); network drop mid-stream surfaces the error while keeping partial text.

8. **Test suite: real and green — verified via CI, not locally.** This Linux audit box has no Swift toolchain and the proxy policy blocks `download.swift.org` (403), so `swift build && swift test` could not be run here. Verified instead through GitHub Actions: the latest `main` run (`b29e3fe`, 2026-07-08) **succeeded** on both jobs — `swift build` + `swift test` on macOS-14, and the full app build + snapshot tests + iOS-simulator UI tests on macOS-15 (`.github/workflows/ci.yml`). The suite is substantive: **230 test functions across 36 files** in `Tests/ReadrKitTests` plus 14 UI/snapshot tests, covering parsers, store back-compat, RAG, providers (mock HTTP), OAuth, PKCE, privacy audit, and context routing. Note: HEAD `5ee8393` (docs/launch prep) postdates that run; its CI was still pending at audit time, but it touches no Swift sources.

---

## Overall verdict

The core reading + annotation product is genuinely solid and crash-safe, and the test suite is real — but **do not launch until the provider-selection lifecycle (🔴-1) and the OpenAI Tier-1 missing-book bug (🔴-2) are fixed, the ChatGPT-subscription button is hidden or beta-labeled (🔴-3), and the README's Readium/SQLite claims are corrected (🔴-4)**. All four fixes are hours, not days.
